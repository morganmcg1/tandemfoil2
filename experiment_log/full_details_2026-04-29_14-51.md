# #1112: Attention dropout=0.1 for OOD slice regularization [MERGED]

## Hypothesis

The baseline `PhysicsAttention` uses `dropout=0.0` throughout — no regularization at the attention or output projection level. With only 1499 training samples and OOD generalization required (geometry and Re holdouts), the model likely overfits its slice token assignments to training geometries. Adding **attention dropout** (`dropout=0.1`) to the `PhysicsAttention` module — both in `F.scaled_dot_product_attention` and in `to_out` — should regularize the slice compression, encouraging more distributed node-to-slice assignments and improving OOD surface-pressure MAE.

Dropout in physics-aware attention models has been shown to improve generalization in fluid simulations (e.g. neural operator literature, FNO ablations). The slice mechanism in Transolver is particularly prone to sharp assignments on training geometries.

## Instructions

In `target/train.py`, change the Transolver instantiation to pass `dropout=0.1`:

```python
model_config = dict(
    space_dim=2,
    fun_dim=X_DIM - 2,
    out_dim=3,
    n_hidden=128,
    n_layers=5,
    n_head=4,
    slice_num=64,
    mlp_ratio=2,
    dropout=0.1,   # <-- add this (was 0.0 implicitly)
    output_fields=["Ux", "Uy", "p"],
    output_dims=[1, 1, 1],
)
```

The `Transolver.__init__` already passes `dropout` through to each `TransolverBlock`, which passes it to `PhysicsAttention`. In `PhysicsAttention.forward`, the dropout is already applied via `dropout_p=self.dropout.p if self.training else 0.0` in `F.scaled_dot_product_attention`, and `self.to_out` wraps `nn.Dropout(dropout)`. So no changes to the model class are needed — just the config dict.

Keep all other hyperparameters at defaults:
- `lr=5e-4`, `weight_decay=1e-4`, `batch_size=4`, `surf_weight=10.0`
- CosineAnnealingLR scheduler, AdamW optimizer
- Transolver: `n_hidden=128, n_layers=5, n_head=4, slice_num=64, mlp_ratio=2`

## Baseline

No prior runs on r4; baseline is `train.py` defaults (`dropout=0.0`). The primary metric to beat is `val_avg/mae_surf_p` (lower is better).

### Bug Found ⚠️

**Two bugs surfaced via this run** (separate to the dropout hypothesis — would affect baseline too):

1. **Data**: `test_geom_camber_cruise/000020.pt` has 761 `inf` values in its ground-truth `y` tensor (at real, non-padded mesh positions). All other test/val samples are clean.
2. **`data/scoring.py:48`**: The masking is implemented as element-wise multiplication `err * surf_mask.unsqueeze(-1).double()`. Because `inf * 0 = NaN` in IEEE 754, any non-finite `y` propagates NaN into the final per-channel sum even though the sample-level `y_finite` skip in the same function is supposed to exclude it. The fix is to use `torch.where(surf_mask.unsqueeze(-1), err, 0.0)` instead. **`scoring.py` is read-only per `program.md`, so I have not patched it** — flagging for advisor attention; happy to open a separate bug-fix PR if you confirm.

# #1128: Per-sample Re-adaptive loss normalization for pressure scale [MERGED]

## Hypothesis

The baseline MSE loss uses global normalization stats (y_mean, y_std from stats.json) that don't account for the ~10–40x variation in per-sample pressure scale across Reynolds number regimes. High-Re samples (Re ~5M, where pressure fluctuations are ~2,000 Pa) contribute ~40x more to the MSE gradient than low-Re cruise samples (Re ~110K, pressure std ~50 Pa). This explains why the current baseline performs poorly on high-Re splits:

- val_single_in_dist: 159.4 (RaceCar single Re 100K–5M)  
- val_geom_camber_rc: 155.6 (RaceCar tandem Re 1M–5M)
- vs val_geom_camber_cruise: 92.9 (Cruise Re 110K–5M)

**Hypothesis:** Normalizing each training sample's squared error by its per-sample pressure standard deviation (computed from the surface pressure nodes) will equalize gradient contributions across Re regimes, leading to better generalization on the high-Re splits and lower val_avg/mae_surf_p.

This is related to "heteroscedastic loss weighting" in the uncertainty estimation literature. By giving each sample equal importance regardless of its pressure dynamic range, the model is forced to learn the pressure distribution shape rather than just fitting the large-magnitude high-Re samples.

Reference: Kendall & Gal, "What Uncertainties Do We Need in Bayesian Deep Learning for Computer Vision?" (NeurIPS 2017) — per-sample uncertainty weighting of loss terms.

## Instructions

Modify the training loop in `target/train.py` to implement per-sample Re-adaptive loss normalization. The key insight is that when computing the volume and surface losses, you should normalize sq_err for each sample by that sample's pressure standard deviation before averaging across nodes.

**Concrete changes inside `train.py`:**

In the training loop where `vol_loss` and `surf_loss` are computed (look for the lines with `vol_mask = mask & ~is_surface`), replace the current per-batch mean with a per-sample normalized mean, then average across samples:

```python
# Current (vulnerable to high-Re gradient domination):
vol_loss = (sq_err * vol_mask.unsqueeze(-1)).sum() / vol_mask.sum().clamp(min=1)
surf_loss = (sq_err * surf_mask.unsqueeze(-1)).sum() / surf_mask.sum().clamp(min=1)

# Replace with per-sample Re-adaptive normalization:
# sq_err shape: (B, N, 3) — last dim is [Ux, Uy, p]
# We normalize by per-sample surface pressure std (channel index 2 = p)
B = sq_err.shape[0]

# Compute per-sample surface pressure std from *predictions* in physical space
# Use the already-computed surf_mask to select surface nodes
surf_p_vals = pred[:, :, 2]  # (B, N) — pressure predictions, normalized space
surf_counts = surf_mask.sum(dim=1).clamp(min=1).float()  # (B,)
surf_p_mean = (surf_p_vals * surf_mask.float()).sum(dim=1) / surf_counts  # (B,)
surf_p_var = ((surf_p_vals - surf_p_mean.unsqueeze(1))**2 * surf_mask.float()).sum(dim=1) / surf_counts
surf_p_std = surf_p_var.sqrt().clamp(min=1e-3)  # (B,) — avoid div-by-zero

# Normalize sq_err per sample by surf_p_std^2 (variance normalization)
# surf_p_std is in normalized space; shape broadcast: (B,) -> (B, 1, 1)
sq_err_norm = sq_err / surf_p_std.pow(2).unsqueeze(1).unsqueeze(2)

vol_loss = (sq_err_norm * vol_mask.unsqueeze(-1)).sum() / (vol_mask.sum().clamp(min=1) * B)
surf_loss = (sq_err_norm * surf_mask.unsqueeze(-1)).sum() / (surf_mask.sum().clamp(min=1) * B)
loss = vol_loss + cfg.surf_weight * surf_loss
```

Note: `pred` here is the model output in normalized space (before `y_std`/`y_mean` rescaling for metric computation). Check where `sq_err` is computed in the existing code to confirm the variable names match — the approach may need minor adjustments to align with the existing tensor variable names.

**Important implementation notes:**
1. The surf_p_std is computed from predictions (not ground truth), so it is always available during both train and eval steps and doesn't leak future information.
2. Use `.clamp(min=1e-3)` to prevent division by near-zero std on boundary-only samples.
3. Keep `cfg.surf_weight = 10.0` unchanged — this re-weighting operates at the sample level, not the loss-component level.
4. After the change, `vol_loss` and `surf_loss` will be dimensionless (normalized), so their logged values will look different from baseline — that's expected. The MAE metrics (computed separately in physical space) are the ground truth for comparison.

That's exactly what we see — high-Re improvements (-14.5%, -3.9%) outweigh low-Re costs (+4.2%, +5.1%) and the average wins.

# #1187: Gradient clipping + raised LR 8e-4 for faster convergence [MERGED]

## Hypothesis

The current training runs time out at ~14 epochs (of 50 configured), meaning the CosineAnnealingLR schedule (T_max=50) barely moves off its initial LR of 5e-4 within the training budget. The model is converging slowly because the effective LR is essentially flat throughout the entire run.

Gradient clipping (`torch.nn.utils.clip_grad_norm_` with `max_norm=1.0`) stabilizes training by bounding gradient norm, which allows safely raising the initial LR from 5e-4 to 8e-4 without divergence risk. A 60% higher LR should make the model converge faster — reaching better minima within the same ~14-epoch budget.

This is a pure convergence speed intervention. The baseline architecture, loss, and all other hyperparameters are unchanged.

## Instructions

Make the following changes to `target/train.py`:

### Change 1: Add gradient clipping in the training loop

After the `loss.backward()` call and before `optimizer.step()`, add:
```python

```

### Change 2: Raise the initial learning rate

Change the `lr` default in the `Config` dataclass (around line 349) from:

to:

## Hypothesis

Adding a dedicated auxiliary MLP head that predicts surface pressure separately — alongside the main 3-channel output head — will improve `val_avg/mae_surf_p` by giving the model a direct, focused training signal for surface pressure without competing gradients from Ux and Uy.

**Rationale:** The current model uses a single linear projection from n_hidden to 3 channels (Ux, Uy, p). The primary metric is surface pressure MAE. This is analogous to auxiliary heads in classification (e.g., GoogLeNet, auxiliary CTC in ASR) where task-specific heads improve gradient flow to shared representations.

# #1116: OneCycleLR (max_lr=2e-3, pct_start=0.3) vs cosine annealing [CLOSED]

## Hypothesis

The baseline uses `CosineAnnealingLR` starting from `lr=5e-4` and decaying monotonically to 0. This schedule is conservative — it never allows the learning rate to rise after the initial epoch, missing the benefit of superconvergence.

**OneCycleLR** (Smith & Topin, 2018 — "Super-Convergence") has been shown to consistently outperform cosine annealing by:
1. Warming up to a high peak LR (`max_lr`) over `pct_start` of training — enabling the optimizer to escape local minima early
2. Aggressively annealing to a very small final LR — enabling fine-grained convergence

We hypothesize that `OneCycleLR(max_lr=2e-3, pct_start=0.3, div_factor=25, final_div_factor=1e4)` will converge faster and reach a better surface pressure optimum than the baseline cosine schedule, since the initial high-LR phase will explore a wider loss basin and the final tight annealing will achieve better surface accuracy.

Paper: https://arxiv.org/abs/1708.07120

## Instructions

**In `target/train.py`, replace the scheduler only.** Keep the AdamW optimizer as-is.

1. **Estimate steps per epoch.** Before creating the scheduler, compute:
   ```python
   steps_per_epoch = len(train_loader)
   total_steps = MAX_EPOCHS * steps_per_epoch
   ```

2. **Replace** the `CosineAnnealingLR` line (around line 407):
   ```python
   # OLD:
   # scheduler = torch.optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=MAX_EPOCHS)
   
   # NEW:
   scheduler = torch.optim.lr_scheduler.OneCycleLR(
       optimizer,
       max_lr=2e-3,
       total_steps=total_steps,
       pct_start=0.3,
       div_factor=25.0,       # initial_lr = max_lr / div_factor = 8e-5
       final_div_factor=1e4,  # min_lr = initial_lr / final_div_factor = 8e-9
       anneal_strategy="cos",
   )
   ```

3. **Move scheduler step to per-batch** (OneCycleLR requires stepping every batch, not every epoch). Find where `scheduler.step()` is called in the epoch loop and move it inside the batch loop (after `optimizer.step()`):
   ```python

   scheduler.step()  # ← inside batch loop, not epoch loop
   ```
   Remove or comment out any `scheduler.step()` call outside the batch loop.

4. **Remove `lr` from Config.** Since OneCycleLR controls its own warmup, the initial LR passed to AdamW should be the OneCycleLR starting point. Set the AdamW `lr` to `2e-3 / 25.0` (i.e., `8e-5`) so it matches `div_factor`:
   ```python

   ```
   (OneCycleLR will immediately raise it to `max_lr` during the warmup phase, so this is just the initialization value.)

# #1115: Oversample surface-node errors by inverse surface fraction [CLOSED]

## Hypothesis

The baseline separates surface and volume nodes into two MSE loss terms weighted by `surf_weight=10.0`. However, surface nodes are **sparse** — on average ~1–2% of total mesh nodes are surface nodes. The effective gradient signal from surface pressure prediction is diluted by the overwhelming number of volume nodes even with the existing surf_weight scaling.

We hypothesize that explicitly **upweighting individual surface-node squared errors by the inverse surface fraction** (i.e., proportional to `N_total / N_surface`) will concentrate gradient signal where it matters most, leading to better `val_avg/mae_surf_p`.

This is mathematically equivalent to computing `surf_loss` as a sample-mean that treats each surface node as if it represents `N_total/N_surface` nodes in the batch. In effect, a surface node's error contributes as much gradient as ~50–100 volume nodes.

## Instructions

**In `target/train.py`, modify the loss computation only.** The key insight: the current `surf_loss` averages MSE over surface nodes with equal weighting. Instead, weight each surface node's squared error by `(N_valid_total / N_surf_total)` — a per-batch scalar that represents how many times more numerous volume nodes are than surface nodes.

**Find the loss block (~lines 438–453)** and replace it with:

```python
# Current node masks
surf_mask = is_surface & mask   # [B, N]  True for real surface nodes
vol_mask = (~is_surface) & mask  # [B, N]  True for real volume nodes

sq_err = (pred - y_norm).pow(2)  # [B, N, 3]

# Count real nodes
n_surf = surf_mask.sum().clamp(min=1).float()
n_vol  = vol_mask.sum().clamp(min=1).float()
n_total = (surf_mask | vol_mask).sum().clamp(min=1).float()

# Surface oversampling factor: how much more numerous are volume nodes
surf_oversample = n_total / n_surf  # scalar, typically ~50–100

vol_loss = (sq_err * vol_mask.unsqueeze(-1)).sum() / n_vol
# Surface nodes weighted by inverse surface fraction
surf_loss = (sq_err * surf_mask.unsqueeze(-1)).sum() / n_surf * surf_oversample

loss = vol_loss + cfg.surf_weight * surf_loss
```

Note: `surf_oversample` is computed per-batch, so it adapts to varying mesh sizes automatically. The `cfg.surf_weight` remains at its default `10.0` — the oversampling factor is additive.

**No other changes.** All other hyperparameters stay at defaults: `lr=5e-4`, `weight_decay=1e-4`, `batch_size=4`, `epochs=MAX_EPOCHS`, CosineAnnealingLR.

# #1114: Curriculum surf_weight ramp 1→20 over training [CLOSED]

## Hypothesis

The baseline uses a fixed `surf_weight=10.0` throughout all training epochs. We hypothesize that this is suboptimal: early in training the model hasn't learned the volume field well, so aggressively penalizing surface pressure immediately may cause unstable gradients or premature over-fitting to surface nodes at the expense of the interior. A **curriculum schedule that ramps surf_weight from 1.0 → 20.0** linearly over training should let the model first establish a coherent global flow field, then increasingly focus on surface pressure accuracy as training matures — yielding better `val_avg/mae_surf_p`.

This follows the "easy-to-hard" curriculum principle widely used in deep learning: start with a simpler objective (roughly equal vol+surf weighting) and progressively tighten the target constraint.

## Instructions

The only change needed is to make `surf_weight` a per-epoch scheduled value rather than a fixed constant.

# #1113: Learnable 3-way domain embedding for multi-domain routing [CLOSED]

## Hypothesis

The dataset spans three physically distinct domains (raceCar-single, raceCar-tandem, cruise) with different mesh geometries, Re ranges, and AoA conventions. While domain information is partially encoded in dims 18–23 of `x` (foil 2 AoA=0 and NACA=(0,0,0) for single-foil; AoA range and gap/stagger differ between raceCar tandem and cruise), the preprocess MLP treats all domains uniformly. Adding a **learnable 3-way domain embedding** added to the preprocessed hidden representation will give the model an explicit routing signal, reducing the learning burden on the slice mechanism.

Domain can be inferred deterministically from the (normalized) input features:
- `x_raw[:, 19:22].sum() == 0` → single-foil (raceCar single)
- `x_raw[:, 19:22].sum() != 0` and `x_raw[:, 14] < 0` (AoA foil 1 < 0 in radians) → raceCar tandem
- Otherwise → cruise tandem

This is a standard conditioning technique used in multi-domain neural operators and is a near-zero cost addition.

## Instructions

In `target/train.py`, add a `DomainEmbedding` module and integrate it into the `Transolver.forward`:

```python
class DomainEmbedding(nn.Module):
    """Learnable 3-way domain tag: 0=racecar-single, 1=racecar-tandem, 2=cruise."""
    def __init__(self, hidden_dim: int):
        super().__init__()
        self.embed = nn.Embedding(3, hidden_dim)
        nn.init.normal_(self.embed.weight, std=0.02)

    def get_domain_ids(self, x_norm: torch.Tensor, x_mean: torch.Tensor, x_std: torch.Tensor) -> torch.Tensor:
        """Infer domain from raw (denormalized) x features. Returns [B] int64 tensor."""
        # Denormalize to get raw features
        x_raw = x_norm * x_std + x_mean  # [B, N, 24]
        # Use first node of each sample (all nodes share same flow conditions)
        foil2_naca = x_raw[:, 0, 19:22]  # [B, 3]
        aoa_foil1 = x_raw[:, 0, 14]      # [B]
        is_single = (foil2_naca.abs().sum(dim=-1) < 1e-6)           # [B]
        is_racecar_tandem = (~is_single) & (aoa_foil1 < 0)          # [B]
        domain_ids = torch.where(is_single,
                        torch.zeros_like(aoa_foil1, dtype=torch.long),
                        torch.where(is_racecar_tandem,
                            torch.ones_like(aoa_foil1, dtype=torch.long),
                            2 * torch.ones_like(aoa_foil1, dtype=torch.long)))
        return domain_ids

    def forward(self, x_norm, x_mean, x_std):
        domain_ids = self.get_domain_ids(x_norm, x_mean, x_std)
        return self.embed(domain_ids).unsqueeze(1)  # [B, 1, hidden_dim]
```

Add `self.domain_embed = DomainEmbedding(n_hidden)` to `Transolver.__init__`.

In `Transolver.forward`, after `fx = self.preprocess(x) + self.placeholder[None, None, :]`, add:
```python
fx = fx + self.domain_embed(x, stats_x_mean, stats_x_std)
```

However, to avoid threading `stats` through the forward call, instead pass the domain tag as part of `data`:

```python
# In the training loop, before model forward:
x_raw_0 = x[:, 0, :]  # [B, 24]
foil2_naca = x_raw_0[:, 19:22]
aoa_foil1 = x_raw_0[:, 14]
is_single = (foil2_naca.abs().sum(dim=-1) < 1e-6)
is_racecar_tandem = (~is_single) & (aoa_foil1 < 0)
domain_ids = torch.where(is_single, torch.zeros_like(aoa_foil1, dtype=torch.long),
             torch.where(is_racecar_tandem, torch.ones_like(aoa_foil1, dtype=torch.long),
             2 * torch.ones_like(aoa_foil1, dtype=torch.long))).to(device)

pred = model({"x": x_norm, "domain_ids": domain_ids})["preds"]
```

In `Transolver.forward`:
```python
def forward(self, data, **kwargs):
    x = data["x"]
    fx = self.preprocess(x) + self.placeholder[None, None, :]
    if "domain_ids" in data:
        fx = fx + self.domain_embed(data["domain_ids"]).unsqueeze(1)  # [B,1,H]
    for block in self.blocks:
        fx = block(fx)
    return {"preds": fx}
```

And simplify `DomainEmbedding` to just `nn.Embedding(3, hidden_dim)` with a direct forward taking domain_ids.

Apply the same domain_ids computation in `evaluate_split` — pass it through `data` dict.

Keep all other hyperparameters at defaults:
- `lr=5e-4`, `weight_decay=1e-4`, `batch_size=4`, `surf_weight=10.0`
- CosineAnnealingLR, AdamW
- Transolver: `n_hidden=128, n_layers=5, n_head=4, slice_num=64, mlp_ratio=2`

Note: domain classification uses raw (unnormalized) x features. The training loop already has access to raw x before normalization — use it there to compute domain_ids.

### Same-epoch comparison (vs partial local baselines)

The two local no-embed baselines I ran were both terminated short (8 and 7 epochs) — they did *not* reach the 14-epoch budget the embed run got. This is anomalous and I cannot explain it (the embed run on the same GPU under the same wall clock reached 14 epochs cleanly). Where the windows overlap:

| Epoch | Baseline 1 | Baseline 2 | Clean embed |
|-------|---:|---:|---:|
| 7 | 167.42 | 190.47 | 156.74 |
| 8 | 165.49 | 179.70 | 152.70 |

So at the *same* training point, the clean embed is ~13–28 points ahead of both partial baselines. But the embed was still bouncing badly even at later epochs (171 / 181 / 135 / 163 / 143 across epochs 10–14), so this lead does not necessarily extrapolate to convergence.

### Why I think this is single-seed noise rather than a real negative
- The val curve fluctuates by 10–30 between adjacent epochs in *both* the embed and the heuristic-embed runs. The 5-point gap between this clean-embed (135.13) and the canonical baseline (129.53) is *smaller* than typical adjacent-epoch jumps.
- The two known measurements of an embed run differ by 4.9 (130.24 heuristic vs 135.13 clean) — the variance between two runs of essentially the same architecture on the same data, on a single seed, is already at the same scale as the gap to the canonical baseline.
- The clean-embed run's *best epoch was 12*, three epochs earlier than the heuristic's best (14). With LR still relatively high at the timeout (cosine T_max=50 means LR≈0.74·5e-4 at epoch 14), neither run has converged — the relative ranking is chiefly governed by which run happens to land on a noise dip just as time runs out.
- Adding a 3×128 embedding is 384 params on a 662k-param model — tiny. The expected effect size is small relative to per-epoch noise.

### What happened
- The implementation is verified correct (100% accurate domain ids, train counts match `program.md` exactly).
- A single-seed comparison gives val 135.13 — does not beat the canonical baseline 129.53, and is also worse than the prior heuristic embed (130.24). Test 3-split (excluding the cruise NaN) tells the same story: clean 132.26 vs heuristic 128.01.
- The expected effect size of a 384-param domain tag is much smaller than the per-epoch noise we see at the 30-min training budget. So the negative single-seed result is **not** strong evidence that the embedding fails — it is roughly consistent with "no detectable effect either way."
- The cleanest finding is that the **manifest-derived lookup itself works** (domain id classification is 100%, vs the heuristic's 68%), so any future experiment that wants per-sample domain conditioning has a clean tag available.

### Suggested follow-ups
1. **Multi-seed comparison.** A 3-seed mean ± std for both `use_domain_embed=False` and `use_domain_embed=True`, on the same fixed seeds. The current 4–5 point single-seed gap is well within plausible per-seed variance, so this is the only way to settle whether the embedding is a wash or actively harmful at this budget.
2. **Different injection strategy.** Adding once after the preprocess MLP gets the embedding diluted by 5 attention blocks. Try (a) concatenating `[fx ; embed]` + linear projection, (b) FiLM-style modulation per Transolver block, or (c) injecting the embedding before *every* block. Higher-leverage uses of the same 384 params.
3. **2-way (single vs tandem) tag.** The largest empirically-observed routing signal is single vs tandem. A 2-way tag isolates that gain from the (apparently weaker) racecar_tandem-vs-cruise distinction.
4. **Longer / different LR schedule.** Both runs hit the timeout with LR still high. Setting `--epochs 14` (so cosine reaches 0 within the budget) would likely tighten convergence and reduce per-epoch noise — which is exactly what's blocking the comparison.
