## Priority 2: Loss Formulation and Signal Quality

### H4: surf_weight=10 + pressure-channel loss upweighting

**Slug:** `pressure-channel-upweight`
**Core idea:** The training loss computes MSE over all 3 output channels (Ux, Uy, p) with equal weight, but the validation metric only cares about surface pressure (channel 2). Upweighting the pressure channel in the loss directly biases the gradient signal toward the metric. Implement via channel-wise weights in `sq_err`: `channel_w = [0.5, 0.5, 2.0]` applied before surf/vol split. This is different from alphonse's log-modulus transform (which changes the loss *space*) — this is a pure weighting change.

**Implementation in train.py training loop:**
```python
# After sq_err = (pred - y_norm) ** 2  [B, N, 3]
channel_w = torch.tensor([0.5, 0.5, 2.0], device=sq_err.device, dtype=sq_err.dtype)
sq_err_cw = sq_err * channel_w[None, None, :]  # broadcast over [B, N, 3]
# Use sq_err_cw in place of sq_err for vol_loss and surf_loss computation
# Keep Re-adaptive sample weighting as-is (applied on top)
```
Add CLI flag `--pressure_upweight float=2.0` (the multiplier on channel 2).

**Expected impact:** 2-5% improvement on val_avg/mae_surf_p (pressure-specific metric). Ux/Uy will slightly degrade, which is acceptable.
**Risk:** Low. Simple multiplicative weight; fully compatible with existing Re-adaptive sample weighting.
**Priority ranking:** #4

### H5: Stochastic depth (drop-path) regularization

**Slug:** `stochastic-depth`
**Core idea:** Stochastic depth (Huang et al., 2016) randomly drops entire residual blocks during training with a linearly increasing probability from 0 (first layer) to p_drop (last layer). It acts as a form of ensemble over network depths, improving OOD generalization. The current Transolver uses standard residual connections in `TransolverBlock`; wrapping each with stochastic depth requires ~10 lines of code. This technique significantly improved ImageNet models and is particularly effective when the model is slightly over-parameterized (which VRAM headroom suggests).

**Implementation:**
Add `StochasticDepth` wrapper module (available in `torchvision.ops` — no new package needed):
```python
from torchvision.ops import StochasticDepth

class TransolverBlock(nn.Module):
    def __init__(self, ..., drop_path_rate=0.0):
        ...
        self.drop_path = StochasticDepth(drop_path_rate, mode="row") if drop_path_rate > 0 else nn.Identity()

    def forward(self, fx):
        # Attention branch
        attn_out = self.attn(self.ln_1(fx), ...)
        fx = fx + self.drop_path(attn_out)
        # FFN branch
        ffn_out = self.mlp(self.ln_2(fx))
        fx = fx + self.drop_path(ffn_out)
        ...
```
In `Transolver.__init__`, set per-block drop_path_rate linearly from 0.0 to `--drop_path float=0.1` across the 5 blocks. Add `--drop_path float=0.1` CLI flag.

**Expected impact:** 1-3% improvement, primarily on OOD splits (geom_camber, re_rand).
**Risk:** Low-medium. torchvision is already available; code change is self-contained.
**Priority ranking:** #5

### H6: Warm-up + cosine decay (LinearWarmup + CosineAnnealing)

**Slug:** `warmup-cosine`
**Core idea:** The current schedule starts at lr=1e-3 immediately (no warmup). With large batches (effectively 4 samples × ~100K nodes = 400K gradient updates per step), the first few gradient steps are taken at full LR, which is high-variance. A short linear warmup (3-5 epochs from lr=1e-6 to lr=1e-3) stabilizes early training and often yields better final convergence. This is different from fern's #1312 (CosineAnnealingWarmRestarts SGDR) — this keeps a single cosine cycle but adds a warm-up prefix.

**Implementation:**
Replace the scheduler in train.py (no new imports needed — use `torch.optim.lr_scheduler.SequentialLR` with `LinearLR` and `CosineAnnealingLR`):
```python
warmup_epochs = 3
warmup_sched = torch.optim.lr_scheduler.LinearLR(
    optimizer, start_factor=1e-3, end_factor=1.0, total_iters=warmup_epochs
)
cosine_sched = CosineAnnealingLR(
    optimizer, T_max=cfg.cosine_tmax - warmup_epochs, eta_min=cfg.cosine_eta_min
)
scheduler = torch.optim.lr_scheduler.SequentialLR(
    optimizer, schedulers=[warmup_sched, cosine_sched], milestones=[warmup_epochs]
)
```
Add `--warmup_epochs int=3` CLI flag.

**Expected impact:** 1-3% improvement. Warmup is generally useful in large-mesh settings with high-variance per-step gradients.
**Risk:** Low. Short warmup (3 epochs) out of 15-epoch budget is acceptable. If fern's SGDR (#1312) merges first, this becomes lower priority.
**Priority ranking:** #6

These tokens act as bottleneck "prototypes" that aggregate and redistribute information across all mesh nodes. For CFD fields with high spatial frequency (boundary layers, wake regions), more slice tokens may be needed to capture the full spatial diversity. Increasing to 96 adds ~50% more prototype capacity with moderate VRAM cost.

## Priority 4: Novel Physics-Informed Directions

### H9: Divergence-free penalty (∇·u ≈ 0 constraint)

**Slug:** `divergence-free-penalty`
**Core idea:** Incompressible CFD solutions satisfy ∇·u = 0 (continuity equation). The current model has no physics constraint — it can predict velocity fields that violate continuity. Adding a soft penalty on the estimated divergence of predicted (Ux, Uy) acts as a physics-informed regularizer. For a 2D mesh, ∂Ux/∂x + ∂Uy/∂z can be approximated using finite differences over the node positions (dims 0-1 of the input x). This forces the model to learn more physically coherent representations.

**Implementation sketch:**
```python
# After getting pred = out["preds"]  [B, N, 3]
# positions: x_phys[:, :, 0:2]  (unnormalized node positions)
# Approximate divergence using centered finite differences over nearest neighbors
# (or simpler: treat Ux/Uy predictions as smooth and compute local gradient via 
#  pairs of near-neighbor nodes found by spatial proximity)
# 
# Simplest viable approach: For each batch, build a kNN graph (k=6) using node
# positions, then approximate ∂Ux/∂x + ∂Uy/∂z using forward differences over
# spatial neighbors weighted by inverse distance.
#
# penalty = (div_approx ** 2).mean() * lambda_div
# loss = vol_loss + surf_weight * surf_loss + lambda_div * div_penalty
```
Add `--lambda_div float=0.01` CLI flag. kNN graph can be precomputed per-sample using `torch.cdist`.

**Expected impact:** 1-3% improvement on OOD splits where the model currently produces non-physical velocity fields. May hurt slightly if kNN approximation is too noisy.
**Risk:** Medium-high. kNN computation adds ~5-10s overhead per epoch. Code complexity is significant. Recommend starting with a very small lambda_div=0.001 and checking that the divergence loss term actually decreases during training.
**Priority ranking:** #9

### H10: Re-conditioned normalization per split domain

**Slug:** `domain-adaptive-norm`
**Core idea:** The global y_std normalization in stats.json averages across all 3 domains (raceCar single, raceCar tandem, cruise), but these domains have ~4× different pressure magnitude ranges. Normalizing by a global std means the model sees very different effective signal scales for different domains. Adding a learned per-domain output scale shift (6 parameters: 3 means + 3 stds, one set per domain) as a FiLM-style affine transform on the final prediction would let the model implicitly re-center per-domain predictions.

**Implementation:**
Domain identity can be inferred from input features (dims 18-23: if all zeros → single-foil raceCar; if nonzero and AoA<0 → tandem raceCar; if AoA>0 → cruise). No manifests needed.
```python
class DomainScale(nn.Module):
    def __init__(self, n_domains=3, out_dim=3):
        super().__init__()
        self.scale = nn.Parameter(torch.ones(n_domains, out_dim))
        self.shift = nn.Parameter(torch.zeros(n_domains, out_dim))

    def forward(self, pred, domain_ids):  # domain_ids: [B]
        s = self.scale[domain_ids]  # [B, out_dim]
        b = self.shift[domain_ids]
        return pred * s[:, None, :] + b[:, None, :]
```
Applied after `mlp2` output. At inference, domain_ids are inferred from input x.

**Expected impact:** 2-5% on OOD splits (geom_camber splits differ most from raceCar single domain). Low parameter count (18 floats).
**Risk:** Medium. Domain classification from features is heuristic and may be imperfect. If thorfinn's re-conditioned output scale head (#1117) covers similar ground, deprioritize.
**Priority ranking:** #10

## Priority 5: Throughput & Training Dynamics

### H11: Gradient accumulation with effective batch_size=8 (bs=2 accumulate=4)

**Slug:** `grad-accum-bs8-v2`
**Core idea:** PR #1230 tried bs=4 + accumulate=2 (effective bs=8) and regressed (+33%). However, that experiment used lr=1e-3 (inherited from bs=4 recipe) — it did not apply linear LR scaling for the larger effective batch. The correct scaling is lr=1e-3 × sqrt(2) = 1.41e-3 for 2× batch size increase. This time: bs=2 accumulate=4 (effective bs=8) with lr scaled to 1.41e-3. Smaller physical batch size also reduces VRAM per step, potentially allowing 2 accumulation steps without hitting 96 GB limit.

**Warning:** PR #1213 tested bs=8 + linear 2e-3 LR and regressed +56%. The key distinction here is sqrt scaling (not linear) plus using accumulation to keep physical bs small for VRAM.

**Implementation:**
```python
# In train.py: track gradient_accumulation_steps
# Add --grad_accum int=4 flag
# Divide total loss by grad_accum before backward
# optimizer.step() and scheduler.step() only after grad_accum steps
# Use lr=1.41e-3 (= 1e-3 * sqrt(2)) with new effective bs=8
```

**Expected impact:** Likely marginal or slightly negative based on prior evidence. Low confidence experiment.
**Risk:** High based on two prior negative results. Only attempt if no higher-priority ideas are available for the slot.
**Priority ranking:** #11 (low priority — only if nothing else is idle)

### H12: FP32 precision on final output + loss (keep AMP for forward pass)

**Slug:** `bf16-loss-fp32`
**Core idea:** The current AMP bfloat16 applies to both the forward pass AND the loss computation. Bfloat16 has only 7 mantissa bits (~0.78% resolution), which may lose precision in the MSE loss computation for extreme Re samples (large pressure values). Computing the loss in fp32 while keeping the forward pass in bf16 is trivial and free:
```python
# After out = model({...})  [computed in bf16]
pred = out["preds"].float()   # cast to fp32 before loss
y_norm_fp32 = y_norm.float()  # cast targets too
# Then compute sq_err in fp32
```
This preserves throughput (the slow part is the bf16 forward pass) while gaining numerical precision in the loss.

**Expected impact:** 0.5-2% improvement, primarily on high-Re samples (val_re_rand). May reduce gradient noise without changing architecture.
**Risk:** Very low. 3-line code change; backward pass is already in fp32 (bf16 only affects forward accumulation).
**Priority ranking:** #12
